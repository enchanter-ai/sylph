## 1. Scope of conformance

### What ISO/IEC 42001 is

ISO/IEC 42001:2023 is the first international management-system standard specifically for AI. It applies the standard ISO management-system structure (Plan-Do-Check-Act, Annex SL) to AI-specific concerns: lifecycle management, transparency, oversight, risk treatment.

### Clauses in scope

| Clause | Title | Scope |
|---|---|---|
| §4 | Context of the organization | In scope |
| §5 | Leadership | In scope (limited — small-team org) |
| §6 | Planning | **In scope, fully** — includes §6.1 risk |
| §7 | Support | In scope |

| §10 | Improvement | **In scope, fully** — includes §10.2 continual improvement |

## 4. Specific clause deep-dives

### §6.1 — Risk management

**Process:**

1. **Identification** — risks surfaced via:
   - `wixie/security-closure/` audit (30 closures found, F-001..F-030)
   - `wixie/inference-engine/` cross-session evidence accumulation (Wald SPRT-elevated patterns)
   - `shared/conduct/precedent.md` per-project failure log
   - `shared/conduct/failure-modes.md` 14-code taxonomy

2. **Analysis** — risks scored via:
   - Severity (critical/high/medium/low) per closure entry
   - Blast radius (single-skill / single-plugin / single-repo / cross-repo / cross-tenant)
   - Wald SPRT log-likelihood ratio for cross-session patterns

3. **Treatment** — driven by:
   - MUST-SHIP queue: ~120h to address F-001, F-002, F-004, F-005, F-010, F-013
   - HIGH-CONFIDENCE queue: ~140h
   - DEFERRED items logged with rationale

**Evidence:** `wixie/prompts/security-closure/results/synthesis.md`

### §8.1 — Operational planning and control

**Lifecycle artifacts per prompt:**

```
prompts/<name>/
├── prompt.<ext>       production prompt
├── metadata.json      model, scores, assertions, version
├── tests.json         regression test cases (≥3, ≥1 edge-case)
├── report.pdf         dark-themed audit (final)
└── learnings.md       hypothesis/outcome log
```

Each lifecycle stage (research → craft → refine → converge → test → harden → translate) is a skill with explicit preconditions, inputs, steps, outputs, and handoff. See `wixie/CLAUDE.md` § Lifecycle.

### §9.1 — Monitoring, measurement, analysis, evaluation

**Monitoring surfaces:**

| Surface | What it captures | Cadence |
|---|---|---|
| `hydra/plugins/audit-trail/state/log.jsonl` | Every tool, hook, policy decision (HMAC-chained) | Real-time |
| `wixie/plugins/inference-engine/state/artifacts.jsonl` | Cross-session emitted patterns | On emit |
| `wixie/plugins/inference-engine/state/catalog.json` | SPRT-reconciled posteriors, LLR, verdicts | On reconcile |
| `wixie/plugins/inference-engine/state/briefings/<plugin>.md` | Top-of-context briefings for next session | Pre-session |
| Per-prompt `learnings.md` | Hypothesis-outcome log | Per iteration |
| `state/precedent-log.md` | Operational gotchas | Per failure |

**Statistical methods used (per `inference-substrate.md`):**
- Wald Sequential Probability Ratio Test (SPRT) for elevation thresholds (LLR > 2.89 elevate; LLR < -2.25 retire)
- Beta-Binomial posteriors over success/failure counts
- Exponential Moving Average decay (half-life)

### §10.2 — Continual improvement

**Mechanism:** the inference-engine substrate. Every plugin emits artifacts on cross-session-relevant failures. The engine reconciles, applies SPRT, and renders per-plugin briefings consumed at session start (top-200-tokens U-curve slot).

**Evidence:** `wixie/plugins/inference-engine/state/briefings/<plugin>.md` files.

**Recursion bound:** depth-1 (substrate watches itself via `substrate-failure` artifacts); depth-2 escalates to human via `inference.escape-valve` file touch.
